# DECISIONS.md

Технічні рішення, які не очевидні з коду і варто зафіксувати, щоб не
передумувати їх кожного разу заново.

## Owner-only аналітика продажів (`GET /api/analytics/summary`)

### Джерело істини: «продано»

`StockMovement.type == MovementType.sale`. Це єдиний тип руху, що означає
факт продажу — пишеться і з `inventory.fulfill` (резерв → продаж), і з
`inventory.sell_direct` (прямий продаж без резерву). Інших шляхів списати
`on_hand` як продаж немає (CLAUDE.md, інваріант №3 — лише `inventory.py`
змінює залишок і завжди пише `StockMovement`).

`sales_count` рахує саме рухи (`StockMovement` рядки), не замовлення —
кожен `fulfill`/`sell_direct` пише рівно один рух, тож «подія продажу» тут
= один рух складу. Якщо колись знадобиться рахувати за замовленнями
(одне замовлення з кількома товарами = одна «подія» для власника) —
агрегувати по `order_id` окремо, це інша метрика.

### Джерело істини: «виручка» (і чому це наближення)

`StockMovement` зберігає лише `delta` (кількість), без ціни чи суми —
схему в цьому кроці НЕ змінюємо. Дослідили обидва можливі снапшоти ціни:

- `StockMovement` — снапшота ціни немає взагалі.
- `Order` / `OrderItem.price_at_order` — є, і це ТОЧНА ціна на момент
  створення замовлення, коли рух прив'язаний до нього.

Тож логіка для кожного продажного руху:

1. Якщо `StockMovement.order_id` НЕ NULL (замовлення з сайту/апки —
   `services/orders.py`: `create_website_order` → `confirm_order`), шукаємо
   точний `OrderItem.price_at_order` за парою (`order_id`, `variant_id`) і
   рахуємо виручку рядка як `qty * price_at_order`. Точно.
2. Якщо `order_id` NULL (ручний резерв «Відклади» з TMA-апки →
   `fulfill`, або гіпотетичний майбутній `sell_direct` без замовлення) —
   точного снапшота немає. Наближаємо як `qty * variant.price`
   (**ПОТОЧНА** ціна варіанта, не ціна на момент продажу).

**Обмеження (свідоме, не помилка):** якщо власник змінив ціну варіанта
ПІСЛЯ ручного продажу без замовлення, виручка за старі періоди для цього
продажу буде показана за новою ціною, не за тією, що була насправді на
момент продажу. Це стосується тільки шляху «Відклади → Продано» в TMA
(без сайту/API) — там немає жодного снапшота ціни в БД, який можна було б
використати замість наближення. Якщо точність тут стане критичною —
рішення: додати `unit_price`/`revenue` напряму в `StockMovement` при
записі продажного руху (зміна схеми, окремий крок, не тут).

`(order_id, variant_id)` неоднозначний, якщо в одному замовленні є два
рядки з однаковим варіантом (теоретично можливо — `create_website_order`
не дедуплікує `items`). Це рідкісний крайній випадок; на практиці
вітрина/кошик не створює дублікатів. Якщо трапиться — береться будь-який
збіг `price_at_order` для цієї пари (без додаткового вирівнювання по qty).

### Вікна періоду

`today` = від 00:00 UTC поточної доби; `7d`/`30d` = rolling (останні N×24
години від моменту запиту, не календарні дні); `all` = без нижньої межі.
Усе на `utcnow()` (UTC, tz-aware) — без урахування таймзони магазину
(її в моделі `Shop` немає).
